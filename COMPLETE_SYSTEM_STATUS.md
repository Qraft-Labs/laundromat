    ## 🔒 BUSINESS LOGIC SAFEGUARDS

    ### Deletion Protection Rules:

    **CUSTOMERS:**
    - ❌ Cannot delete if they have UNPAID orders
    - ❌ Cannot delete if they have PARTIAL payment orders
    - ✅ Can delete ONLY if: No orders OR all orders fully PAID

    **ORDERS:**
    - ❌ Cannot delete if payment_status is UNPAID
    - ❌ Cannot delete if payment_status is PARTIAL
    - ✅ Can delete ONLY if: payment_status is PAID (fully settled)

    **USERS (Desktop Agents/Managers):**
    - ❌ Cannot delete if they created orders with pending dues
    - ❌ Cannot delete if they are assigned to active employees
    - ✅ Can delete ONLY if: No financial obligations tied to them

    **PRICE ITEMS:**
    - ❌ Cannot delete if used in ANY order (historical data)
    - ✅ Can mark as `is_active = false` instead (soft delete)

    **EMPLOYEES:**
    - ❌ Cannot delete if they have salary payment history
    - ✅ Can mark as `employment_status = 'TERMINATED'` instead

    ## 📱 REAL-WORLD OPERATIONS

    ### Daily Workflow:

    1. **Desktop Agent logs in** → john.agent@lushlaundry.com
    2. **Customer brings clothes** → Search existing or create new customer
    3. **Create order** → Select items (Shirt, Trouser), choose service (wash/iron)
    4. **Backend calculates** → Subtotal, discount, total (server-side)
    5. **Accept payment** → Cash, Mobile Money, or Bank Transfer
    6. **Print receipt** → Order number: ORD20260123
    7. **Clothes processed** → Washers and ironers handle garments
    8. **Order status updated** → pending → processing → ready → delivered
    9. **Customer pickup** → Pay balance if partial payment
    10. **Revenue recorded** → Real-time dashboard updates

    ### Monthly Accounting:

    - **Payroll:** UGX 6.2M to 12 employees (via salary_payments table)
    - **Expenses:** Detergents, electricity, water (via expenses table)
    - **Revenue:** Automatically calculated from orders (via financial_summary)
    - **Reports:** Income statement, balance sheet, trial balance
    - **Inventory:** Track stock levels, reorder when below minimum
